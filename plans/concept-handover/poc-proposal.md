# POC Proposal — A Runnable Skeleton for the ALM Ontology Layer

### Proof-of-concept plan for the three-layer architecture, with a report-generating CLI

> **Status:** proposal for review. This is exploration, not precise development. The goal is to make the [handover's](alm-knowledge-graph-handover.md) three layers tangible end-to-end and to produce an interactive HTML *report artifact* that shows what the ontology actually buys us — coverage gaps, safety-level propagation, and traceability — over a small, fully-fictional aerospace example.

## 1. What this POC is (and is not)

This is the "runnable skeleton" called out in the handover's [Section 12](alm-knowledge-graph-handover.md): a LinkML model, a loader, verified in-memory inference functions, and tests that assert the coverage gap and the propagation result — exercised end to end.

It demonstrates the **model → regenerable graph view → tooling** spine and its discipline: *tables are truth, the graph is a regenerable view, the model is the schema.*

**In scope**
- A single declarative **LinkML model** (Layer 0) that generates Python types, SQL DDL, and docs.
- A **committed, LLM-authored example** for the fictional VM-E1 aircraft under `data/{requirements,architecture,defects}/` (see Section 4 for the authored, traceable structure).
- A loader that builds the durable substrate (**SQLite + Parquet**) from `data/`.
- **Two concrete graph realizations over the same tables**, both first-class:
  - **DuckPGQ** (SQL/PGQ) as the *portable graph contract* — Layer 1;
  - **rustworkx** as the *in-memory, auditable inference* engine — Layer 2.
- DuckDB anti-join queries for completeness/coverage (Layer 2 analytical core).
- The canonical questions answered as **fixed CLI verbs** under one global Typer app `almon`: `coverage`, `impact`, `propagate`.
- A **cross-engine agreement check** — `impact` computed via DuckPGQ and via rustworkx must match, proving the graph is a faithful regenerable view.
- A **report CLI** that writes:
  - a gitignored Markdown report `.report/<date>/<topic>-<HHMM>.md` — narrative + tables, no embedded charts;
  - a self-contained interactive **HTML** report `.report/<date>/<topic>-<HHMM>.html` — Plotly charts plus an interactive traceability graph laid out from rustworkx.
- An optional `serve` verb (stdlib `http.server`) to browse `.report/` from localhost.

**Out of scope (intentionally)**
- Vector / recall layer (Typesense, LanceDB) — kept separate and pipelined per the handover; not needed to show ontology value.
- The governance/provenance plane from the [deep-research critique](deep-research-report.md) (OpenLineage, MLflow, Sigstore, lakeFS) — a valid later layer; would bloat an exploration POC.
- MCP server surface — the CLI verbs *are* the fixed contract here; wrapping later is mechanical.
- Codebeamer ingestion — upstream and out of scope; we ship a curated authored example instead.
- The RDF/SPARQL standards port — held in reserve (a future `--rdf` export only).

## 2. Domain note: DAL, not ASIL

The handover's worked slice is automotive and uses **ASIL** (ISO 26262). The example domain you chose is **aerospace**, so this POC uses the domain-correct vocabulary: **DAL — Design Assurance Level — A through E** (DO-178C / ARP4754A), where **A** is most critical (catastrophic) and **E** is no-effect.

The *mechanism is identical* to the handover's ASIL examples — a criticality level that (a) propagates down the architecture composition and (b) requires a qualifying passing test for critical items. Only the controlled vocabulary changes (`{A,B,C,D,E}` instead of `{QM,A,B,C,D}`). This is the single deliberate divergence from the handover, made for domain credibility; it is called out here so a reviewer expecting "ASIL" knows why it reads "DAL."

## 3. How it maps to the handover's three layers

| Handover layer | POC realization | Library |
|---|---|---|
| **Layer 0 — the model (spine)** | One YAML schema → generated Pydantic classes, SQL DDL, Markdown docs | `linkml` |
| **Substrate (tables are truth)** | Committed authored YAML in `data/` → built SQLite + Parquet | `sqlite3` (stdlib), `pyarrow` |
| **Layer 1 — portable graph contract** | Property graph declared **as a SQL/PGQ view over the tables** | `duckdb` + **DuckPGQ** |
| **Layer 2 — analytical core** | Anti-joins / coverage / completeness over tables & Parquet | `duckdb` |
| **Layer 2 — in-memory inference** | DAL propagation + variable-length traversal as auditable, unit-tested code | `rustworkx` |
| **Layer 2 — standards port (reserve)** | *Not built.* Noted as a future `--rdf` export only | (deferred) |

The constraint split is preserved: **structural** validation (required fields, DAL vocabulary, cardinalities) runs cheaply against the generated Pydantic types at load; **cross-entity completeness** ("every DAL-A requirement has a passing verifying test") is a DuckDB anti-join.

DuckPGQ and rustworkx are both rebuilt from the tables on each run — neither is a store of record. They are the two halves of the handover's "regenerable view": the portable standards-based contract (SQL/PGQ) and the auditable imperative engine (rustworkx), kept honest against each other by the cross-engine agreement check.

## 4. The committed example — authored, traceable, not randomized

The example is **LLM-authored** (during POC build), structured for traceability, and committed under `data/`. The three folders carry distinct meanings:

### `data/requirements/` — the binding specification (the *what*)
The contractual statement of expected outcomes for the VM-E1: what the aircraft and its software/systems must achieve. Each requirement carries a **DAL**, acceptance criteria, and `refines` links forming decomposition chains (system → subsystem → component-level). This is the source of truth for "expected outcome." Test cases that `verify` requirements (carrying a passed/failed/not_run outcome) are co-located here, since they assert the binding spec.

### `data/architecture/` — the technical breakdown (the *how*) — **authored from requirements, never auto-generated**
The engineering design that satisfies the requirements, authored by LLM generation *from* the requirements:
- **features and components**, with their `composed_of` decomposition;
- **component diagrams** (Mermaid) showing structure and interfaces;
- **interaction / sequence diagrams** (Mermaid `sequenceDiagram`) showing key behaviours (e.g. throttle command → motor-controller → motor; BMS fault → flight-computer → pilot alert);
- the **allocation** traceability (`allocated_to`: requirement → component).

This is genuine design work and a writing task — it cannot be derived mechanically from the requirements, so it is authored and committed as Markdown (with embedded diagrams) plus structured YAML for the machine-readable elements/edges.

### `data/defects/` — problems and violations (the *what went wrong*)
Authored defects, each describing a problem that **violates one or more requirements** and **occurs in one or more affected architecture components** (`affects`). Every defect references the requirement(s) it breaks and the component(s) where it manifests, so impact analysis has something real to traverse.

A seed governs only incidental fields (ids, dates, owners) for reproducibility; the semantic content is authored. `faker` is used only for that incidental metadata, never for requirement/architecture/defect text.

### Entities & relationships (the LinkML model)
- `Requirement` (`dal` ∈ {A,B,C,D,E}), `ArchitectureElement`, `TestCase`, `Defect`.
- `refines` (req→req, transitive), `allocated_to` (req→element), `composed_of` (element→sub-element, transitive), `verifies` (test→req, carries outcome), `affects` (defect→element).

`verifies` is the "edge that carries a property" shape: an association class in the model, an edge payload in rustworkx, an edge table in DuckPGQ, an outcome column in the tables.

### The three canonical questions → three CLI verbs
1. **`coverage`** — which DAL-A (or higher-criticality) requirements have no *passing* verifying test. Positive pattern from the graph; the gap from a relational anti-join (DuckDB).
2. **`impact`** — from a requirement, through its allocation, down any depth of architecture composition, to the defects touching those elements. A variable-length path — computed on **both** DuckPGQ and rustworkx and asserted equal.
3. **`propagate`** — the inferred *effective* DAL of each architecture element, by propagating the strongest contributing level down the composition. Explicit, unit-tested rustworkx code (the "reasoner as auditable code" example). Conservative/monotonic, as the handover cautions — analysis only, never certifiable.

## 6. CLI surface — one global Typer app, `almon`

A single Typer application with subcommands; entry point `almon` (for **alm-on**tology). Mirrors the handover's "tools are verbs" principle.

```bash
# Layer 0 — regenerate artifacts from the model
uv run almon model gen             # alm.yaml -> Pydantic + SQL DDL + docs

# Build the durable substrate from the committed authored data/
uv run almon build                 # -> data/warehouse/alm.sqlite + data/warehouse/parquet/*.parquet
                                   #    (loads + structurally validates against generated types)

# Validation (structural + cross-entity)
uv run almon validate              # Pydantic structural + DuckDB completeness

# The canonical questions
uv run almon coverage --min-dal A
uv run almon impact --req REQ-0007 --max-depth 6 --engine both   # rustworkx | duckpgq | both
uv run almon propagate

# Reporting (writes .md + interactive .html under .report/<date>/)
uv run almon report --topic coverage
uv run almon report --topic full   # all questions in one report

# Browse reports from localhost (convenience; HTML is already interactive as a file)
uv run almon serve --port 8000
```

`impact --engine both` runs DuckPGQ and rustworkx and reports agreement (the same check `test_cross_engine.py` enforces). There is no separate `generate` verb in the core flow: the example is authored and committed; a thin internal helper may (re)assemble incidental fields deterministically under `--seed`, but architecture content is authored, not generated.

## 7. The report artifact

**Path convention (gitignored):**
```
.report/
└─ 2026-06-20/
   ├─ coverage-1432.md       # Markdown: narrative + tables, links to the .html
   └─ coverage-1432.html     # self-contained, interactive (Plotly), opens as a file
```

**Markdown (`.md`)** — narrative summary plus tables (`pandas.DataFrame.to_markdown()`): e.g. uncovered DAL-A requirements with allocations, test-outcome counts, effective-DAL per element, top affected elements. A relative link points to the `.html` for interactive charts and graph. No embedded charts, no PNGs.

**HTML (`.html`)** — a single self-contained file (Plotly's JS inlined once) containing:
- **Charts** (Plotly, interactive): coverage by DAL level (bar), test-outcome distribution (stacked bar), defects-per-element (histogram), effective-vs-declared DAL.
- **Graph** (interactive): the traceability/impact subgraph (requirement → architecture → defects), laid out with `rustworkx` layout (e.g. `rustworkx.spring_layout`) and drawn as Plotly node/edge scatter traces — hover to inspect, zoom, pan.

Plotly HTML is interactive when opened directly as `file://`, so **no server is required**. `serve` (stdlib `http.server` rooted at `.report/`, with a generated index by date/topic) is a convenience.

## 8. Dependencies (uv-managed)

Core:
- `typer` — global CLI / verb contract
- `linkml` — Layer 0 schema → Pydantic / SQL DDL / docs
- `pydantic` — generated types + structural validation
- `duckdb` — analytical core + host for DuckPGQ (pin the version)
- `rustworkx` — in-memory regenerable graph, auditable inference, graph layout
- `pyarrow` — Parquet I/O
- `pandas` — table shaping for reports
- `plotly` — interactive charts **and** interactive graph rendering (the only viz dep)
- `faker` — incidental metadata only (ids/dates/owners), not semantic content

**DuckPGQ** is a DuckDB *community extension*, not a pip package. It is loaded at runtime in `warehouse.py`:
```sql
INSTALL duckpgq FROM community;
LOAD duckpgq;
```
Because the deep research flagged it operationally maturing, the code pins the DuckDB version, verifies the extension loads, and falls back to rustworkx (with a clear message) if a given platform/version can't load it — so the POC always runs.

Dev:
- `pytest` — golden-fixture + cross-engine tests
- `ruff` — lint/format

## 10. Build phases

1. **Scaffold** — `pyproject.toml`, uv venv, package skeleton, `.gitignore`, README disclaimer banner. `almon --help` works. *(walking skeleton)*
2. **Layer 0** — author `alm.yaml` (entities, DAL enum, relationships); wire `model gen` → Pydantic + DDL + docs.
3. **Author the example** — write the committed VM-E1 content:
   - `requirements/` binding spec (DAL distribution, refine chains, acceptance criteria) + co-located test cases;
   - `architecture/` breakdown authored *from* the requirements — components + `composed_of`, component diagrams, sequence diagrams, `allocated_to`;
   - `defects/` violations referencing requirements + affected components.
   Seed in **deliberate** findings so reports show something: one uncovered DAL-A requirement, one failed-only verification, one defect on a deep sub-element. Commit `data/`.
4. **Substrate** — `build` loads `data/` → `data/warehouse/` SQLite + Parquet, validating against the generated Pydantic types.
5. **Validation** — `validate`: Pydantic structural pass + DuckDB completeness anti-join.
6. **rustworkx engine** — `graph_rustworkx.py`: regenerable view, `propagate` (DAL), `impact` (variable-length path). Unit-test against golden mini-graphs.
7. **DuckPGQ engine** — `graph_duckpgq.py`: declare the property graph as a SQL/PGQ view over the tables; `impact` via `MATCH`. Add `test_cross_engine.py` asserting DuckPGQ `impact` == rustworkx `impact`.
8. **Queries** — `coverage` anti-join in DuckDB; reconcile graph-positive vs relational-negative.
9. **Reporting** — `report.py` + `charts.py`: Markdown tables/narrative + interactive HTML (Plotly charts + rustworkx-laid-out graph) under `.report/<date>/`.
10. **Serve** — `serve.py` lists and serves reports.
11. **(Reserve)** RDF/SPARQL export stub if/when standardized interchange is wanted.

Phases 1–9 are the meaningful core; 10–11 are nice-to-have.

## 11. What "done" looks like for the POC

- `uv run almon build && uv run almon report --topic full` produces a dated report under `.report/` with: a DAL-coverage table flagging the seeded uncovered requirement, interactive Plotly charts, and an interactive impact graph that opens straight from the file.
- `pytest` is green: propagation golden test, coverage anti-join test, and **cross-engine agreement** (DuckPGQ `impact` == rustworkx `impact`) all pass.
- The graph is provably a *view*: deleting `data/warehouse/` and re-running `build` reproduces identical reports — nothing lives only in either graph engine.
- The model is the spine: editing `alm.yaml` + `model gen` changes the generated types/DDL/docs together.
- Reproducibility: `data/` is committed (LLM-authored, seed 42 for incidental fields); anyone can rebuild the warehouse and reports from it.
